В Kotlin Coroutines **Flow** — это мощный инструмент для работы с асинхронными потоками данных. 
Чтобы понять разницу между «холодными» (Cold) и «горячими» (Hot) потоками, 
проще всего представить их как **фильм в онлайн-кинотеатре** против **трансляции по радио**.

---

## 1. Холодные потоки (Cold Flow)

Это стандартный `flow { ... }`. Он **неактивен**, пока на него кто-то не подпишется (не вызовет `collect`).

* **Принцип работы:** Код внутри блока `flow` запускается заново для каждого нового коллектора.
* **Аналогия:** Netflix. Фильм не идет, пока вы не нажмете «Play». Если два человека начнут смотреть один и тот же фильм в разное время, каждый увидит его с самого начала независимо друг от друга.
* **Пример:** Запрос в базу данных или сетевой вызов.

```kotlin
val coldFlow = flow {
    println("Поток запущен")
    emit(1)
    emit(2)
}

// Пока не вызовем collect(), ничего не произойдет.
// Если вызовем дважды — "Поток запущен" напечатается дважды.

```

## 2. Горячие потоки (Hot Flow)

Эти потоки **активны всегда**, независимо от того, слушает их кто-то или нет. Они существуют в памяти как независимые объекты.

* **Принцип работы:** Данные рассылаются всем подписчикам одновременно. Те, кто подписался позже, пропускают то, что было выдано ранее.
* **Аналогия:** Радио или прямой эфир. Если вы включили радио в 10:15, вы не услышите новости, которые были в 10:00.
* **Основные типы:** `SharedFlow` и `StateFlow`.

### StateFlow

Специализированный горячий поток, который всегда хранит **последнее состояние**.

* Всегда имеет начальное значение.
* Новый подписчик сразу получает текущее состояние.
* Идеален для UI (вместо `LiveData`), так как сохраняет данные при повороте экрана.

### SharedFlow

Более гибкий горячий поток для рассылки событий.

* Не хранит состояние по умолчанию (но можно настроить `replay`).
* Используется для разовых событий: навигация, показ SnackBar, алерты.

---

## Основные различия

| Характеристика | Cold Flow (`flow`) | Hot Flow (`State/Shared`) |
| --- | --- | --- |
| **Активность** | Ленивый (запуск при подписке) | Активный (всегда работает) |
| **Хранение данных** | Не хранит | Хранит (StateFlow) или буферизирует |
| **Количество подписчиков** | 1 подписчик на 1 запуск (Unicast) | Много подписчиков (Multicast) |
| **Ресурс** | Выполняет код каждый раз | Выполняет код один раз, делит результат |

---

### Когда что использовать?

1. **Cold Flow:** Для разовых операций получения данных. Например, получить список пользователей из Room или совершить один API-запрос.
2. **StateFlow:** Для хранения состояния экрана во `ViewModel`. UI просто подписывается и всегда знает, что рисовать.
3. **SharedFlow:** Для событий «выстрелил и забыл», которые должны получить все активные экраны (например, уведомление об ошибке авторизации).
